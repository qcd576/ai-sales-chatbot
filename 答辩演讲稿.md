# 销售（售后）服务AI聊天机器人开发项目 —— 答辩演讲稿

**答辩人：乔成铎 | 时长：约6-7分钟**

---

各位老师好！我是乔成铎，我们小组做的项目是「销售与售后服务场景的AI聊天机器人全栈系统」，以「智选科技」这个虚构的消费电子品牌为应用背景。接下来我将从项目背景、技术架构、AI核心实现、系统开发、部署测试和成果总结六个方面进行汇报。

## 一、项目背景与市场分析（约1分钟）

首先看行业背景。在电商销售和售后客服领域，传统人工客服面临几个核心痛点：人力成本随业务规模线性增长、高峰期响应速度慢、服务质量受情绪和知识水平影响参差不齐、产品知识难以全面覆盖、以及无法真正实现24小时不间断服务。

针对这些痛点，我们设计了这套AI客服系统。核心思路是：不满足于"简单调用大模型API"，而是通过系统化的提示词工程、结构化知识库和检索增强生成技术，构建一个真正专业、可控、可维护的垂直场景AI客服。

目标用户画像覆盖电商消费者、潜在购买者、已购用户和投诉用户，年龄跨度18到60岁。核心功能包括五大块：产品咨询、订单查询、退换货政策解答、故障排查引导和客户情绪安抚。

整个项目周期为10周，按照软件开发生命周期的完整流程推进。

（**翻到第4页**）

## 二、技术选型与系统架构（约1分钟）

系统采用前后端分离的四层架构。

前端展示层使用 React 18 + TypeScript + Vite 6 构建，并利用 Capacitor 将 Web 应用打包为 Android APK。后端服务层采用 Python FastAPI 异步框架，这是目前性能最好的 Python Web 框架之一。AI 模型层对接 DeepSeek Chat API，这是我们项目最核心的技术选型——选择 DeepSeek 是因为它在中文理解上表现优异，API 接口兼容 OpenAI 格式，而且性价比极高，推理成本仅为国际同类模型的十分之一到五十分之一。数据持久层使用 SQLite 3 + SQLAlchemy ORM，三张核心表覆盖会话、消息和知识库。

架构的数据流向是：用户通过浏览器或 APP 发送消息，经过 HTTP/REST API 到达 FastAPI 后端，后端一方面调用知识库检索获取相关内容，另一方面构建包含系统提示词和知识上下文的完整消息列表，发送给 DeepSeek API，最后将 AI 回复返回给用户并持久化存储。

（**翻到第5页**）

## 三、AI功能核心实现（约1.5分钟）

这是本项目最大的技术亮点，也是区别于"简单调用API"的关键差异化设计。

我们设计了一套**六层系统提示词框架**，从底层到顶层分别是：角色设定层、品牌知识注入层、能力范围层、服务原则层、边界处理层和知识增强层。这六层协同工作，确保大语言模型始终在预期的行为框架内进行对话。

每一层解决一类控制需求：最底层告诉模型"你是谁"，第二层注入品牌背景和产品信息，第三层明确你能做什么，第四层规范怎么做——要专业准确、热情友好、有销售意识但不强推，第五层处理边界——遇到非业务问题怎么引导、遇到投诉怎么安抚，最顶层注入知识库检索结果作为参考。

举个例子，如果用户说"你们的产品质量太差了"，普通调用API可能回一句"有什么问题您可以反馈"，但我们的系统会先共情——"很抱歉给您带来不好的体验"，然后了解具体问题，再积极提供解决方案。这就是提示词工程的价值。

此外，我们构建了覆盖7个分类共32条Q&A对的结构化知识库，开发了基于关键词匹配评分的检索算法——问题匹配权重3倍、答案匹配权重1倍、完整短语匹配额外加分——取Top-3最相关条目注入上下文。对比测试表明，启用RAG后在回复精确性、品牌一致性、价格准确度和政策准确性四个维度上都有显著提升。

（**翻到第7页**）

## 四、全栈系统开发（约1分钟）

后端方面，我们使用 FastAPI 实现了三大模块共10个 RESTful API 端点：聊天模块负责消息发送和 AI 回复，会话模块支持创建、列表、消息历史和删除，知识库模块提供完整的 CRUD 接口。

几个关键技术特性值得提一下。第一，采用异步架构——async/await 配合 httpx.AsyncClient，与 FastAPI 原生异步模型完美兼容。第二，实现了自动降级机制——当后端 API 不可用时，自动切换到内置的离线智能回复引擎，覆盖20多个关键词和模糊匹配逻辑，保证用户无感知切换。第三，通过 Swagger UI 自动生成交互式 API 文档，方便调试和对接。

前端方面，我们开发了三个核心页面：用户聊天页面 ChatPage、服务器设置页面 SetupPage 和管理后台 AdminPage。聊天页面支持欢迎引导、多轮对话、快捷问题和离线模式。管理后台支持知识库条目的增删改查、分类筛选和关键词搜索。前端还实现了 PWA 支持和全局错误边界，防止白屏崩溃。

（**翻到第9页**）

## 五、系统测试与云端部署（约1分钟）

测试方面，我们覆盖了基础对话、产品咨询、订单物流、退换货、售后保修、边界处理和离线降级等十大功能场景。测试结果显示，10个API端点全部通过，DeepSeek API响应时间小于3秒，知识库RAG匹配准确率超过85%，离线模式20多个关键词全部命中。

开发过程中也遇到了几个有代表性的问题。比如Windows下Python脚本的中文GBK编码错误、Android Gradle构建时中文路径报错、国内网络下载Gradle超时——通过替换腾讯云镜像解决、以及阿里云安全组和UFW双重防火墙导致端口无法访问——需要在两边分别配置入站规则。

部署方面，系统已部署到阿里云ECS云服务器，公网地址是 120.27.247.44:8000。部署采用了"一机全栈"方案——单台服务器通过 Uvicorn 同时承载后端 API 服务和前端静态资源托管。同时使用 Capacitor 将 Web 应用打包为 Android APK，仅4.1MB，内置离线引擎，安装即用。

（**翻到第11页**）

## 六、项目总结（约30秒）

最后做一个总结。本项目以智选科技消费电子品牌为背景，完成了从市场调研、架构设计、AI核心实现、全栈开发到云端部署的完整软件开发生命周期。项目有五个核心创新点：六层提示词工程框架、RAG检索增强生成、全自动离线降级、一机全栈部署和Web+APP双端交付。

项目的Web应用、APK下载、API文档和GitHub开源代码仓库均已在线上，各位老师可以随时访问体验。
