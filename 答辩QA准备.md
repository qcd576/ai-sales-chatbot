## Q1: 这个项目你是怎么做的？具体流程是什么？

按照软件开发生命周期完整流程推进，共10周：

1. **市场调研**（第1-2周）：分析电商客服痛点，确定差异化定位——不满足于简单调用API，而是通过提示词工程+RAG构建真正专业可控的垂直AI客服
2. **架构设计**（第3-4周）：确定前后端分离四层架构、技术选型（React+FastAPI+DeepSeek+SQLite），设计数据库ER图
3. **AI核心实现**（第5-6周）：构建六层提示词框架、32条知识库、DeepSeek API对接、RAG检索算法。这是投入精力最多的阶段
4. **全栈开发**（第7-8周）：后端10个API端点 + 前端3个页面 + 数据库3张表，持续联调
5. **测试部署**（第9-10周）：功能测试、阿里云ECS部署、Capacitor打包APK

> 追问：你负责了哪些部分？
> 答：我负责了项目整体架构设计、AI核心实现（提示词工程+知识库+RAG）、后端API开发和云端部署。

## Q4: 为什么选择 DeepSeek 而不是其他大模型？

1. **中文能力优秀**: DeepSeek在中文理解和生成任务上表现优异，适合中文客服场景
2. **性价比极高**: API推理成本约为国际同类模型（GPT-4等）的1/10到1/50，适合商业规模化部署
3. **API兼容性**: 接口兼容OpenAI Chat Completion格式，迁移成本低
4. **课程契合度**: 国产大模型，契合"大数据机器学习"课程国产化技术应用导向

> 追问：如果DeepSeek挂了怎么办？
> 答：系统设计了自动降级机制。后端检测到API不可用时，自动切换到内置离线引擎（20+关键词匹配+模糊匹配），用户无感知。APK版本更是完全内置离线引擎。

## Q5: 你这个和直接调用大模型API有什么区别？

**这是最核心的差异化点**。直接调用API就像让一个什么都不知道的人去当客服——他会说话但不懂你的产品和政策。

我们做了三层增强：

1. **六层提示词工程**：不是写一句"你是一个客服"，而是从角色设定、品牌知识、能力范围、服务原则、边界处理到知识增强，逐层精确控制模型行为。比如遇到投诉场景，普通调用可能冷冰冰回复，我们的系统会先共情、再了解问题、再提供方案

2. **RAG检索增强生成**：构建了32条结构化知识库，用户提问时先检索最相关的3条知识，注入到提示词中。这样回复基于真实品牌数据而非模型"编造"——价格、政策100%准确

3. **离线降级机制**：API不可用时自动切换内置引擎，保证服务不中断

## Q6: 遇到的最大困难是什么？怎么解决的？

**最大的困难是提示词工程的迭代调优**。

一开始，模型回复要么太啰嗦（300字限制不生效），要么在非业务问题上放飞自我（讲笑话、写代码什么都能干），要么销售意图太强（用户问个参数就疯狂推销）。

解决过程：3轮实战测试→每轮分析不符合预期的回复→定位是哪一层提示词失效→调整该层内容→重新测试对比。比如边界处理层，最初写的是"不要回答非业务问题"，模型会生硬拒绝；改成"先道歉+解释能力范围+主动引导回业务场景"后，用户体验明显提升。

**其他困难**：Windows下GBK编码错误（强制UTF-8）、Android中文路径报错（overridePathCheck）、Gradle国内下载超时（腾讯云镜像）、阿里云双重防火墙（分别配置UFW和安全组）。

## Q7: 你的提示词工程具体是怎么设计的？

设计了一套**六层系统提示词框架**，从底层到顶层逐层递进：

| 层级 | 内容 | 作用 |
|------|------|------|
| ①角色设定 | "你是智选科技AI客服小智" | 身份锚定 |
| ②品牌知识 | 品牌定位、五大产品线、价格区间 | 领域知识注入 |
| ③能力范围 | 六大服务场景清单 | 行为边界约束 |
| ④服务原则 | 专业/热情/耐心/销售意识/情绪安抚 | 回复风格控制 |
| ⑤边界处理 | 非业务/敏感/转人工的具体话术模板 | 安全兜底 |
| ⑥知识增强 | 动态注入RAG检索到的Top-3知识 | 事实准确性保证 |

关键设计理念：**分层控制，协同生效**。每一层解决一类控制需求，六层同时工作确保LLM在预期框架内对话。这个框架和"写一段prompt"的本质区别在于系统性和可维护性——改边界规则不需要动品牌知识。

## Q8: RAG检索是怎么做的？为什么不用向量检索？

**关键词匹配评分算法**：

1. 用户输入 → 中文分词 → 提取关键词
2. 遍历知识库32条Q&A → 对每条计算匹配分数：
   - 问题匹配权重 ×3（出现在问题中说明相关性高）
   - 答案匹配权重 ×1
   - 完整短语匹配额外加分（+10问题，+5答案）
3. 按分数降序 → 取Top-3 → 构建上下文 → 注入System Prompt

**为什么没用向量检索？**
课程项目规模下（32条知识库），关键词匹配更简单直接、可解释性强、无额外依赖（不需要embedding模型或向量数据库）。但报告中已说明这是未来优化方向——当知识库扩展到数百条时，向量检索的语义匹配优势会体现。

## Q9: 数据库为什么选SQLite而不是MySQL？

1. **零配置**: 不需要独立数据库服务，SQLite引擎嵌入应用进程
2. **适合规模**: 课程项目单机部署，并发量低，SQLite完全够用
3. **可迁移**: SQLAlchemy ORM屏蔽了底层差异——未来升级到PostgreSQL只需改连接URL，上层代码不动
4. **便携性**: 数据库是单个文件（44KB），备份和迁移极其方便

## Q10: 你的APK是怎么做的？和Web版有什么区别？

使用**Capacitor**将Web应用打包为原生Android APK：

- Web版通过Vite构建为静态HTML/JS/CSS
- Capacitor将构建产物封装到Android WebView中
- Gradle编译生成APK安装包（4.1MB）

**APK与Web版的主要区别**：
- APK跳过设置页，打开直接进入聊天（更适合移动端体验）
- APK默认使用离线模式，内置20+关键词智能回复引擎
- 纯内联样式确保WebView渲染兼容（TailwindCSS在WebView中有兼容问题）
- 也可以手动配置服务器地址连接后端使用DeepSeek AI

## Q11: 这个项目实际能用在商业场景吗？还缺什么？

**已有能力**：基础的产品咨询、政策解答、故障排查引导可以在消费电子电商场景中使用。

**商业化的关键差距**：

1. **知识库规模**：32条远不够，需要扩充到数百条覆盖更细分的场景
2. **向量检索**：用embedding替代关键词匹配，提升语义理解准确性
3. **多轮对话管理**：当前支持历史上下文但不支持槽位填充等复杂对话管理
4. **安全加固**：需要JWT认证、Rate Limiting、HTTPS、SQL注入防护
5. **多模型支持**：当前仅对接DeepSeek，商业场景需要模型热切换能力
6. **数据分析**：缺少对话数据分析和用户行为洞察

报告中"结论与展望"章节已详细列出这些方向。

## Q12: 你从这个项目中学到了什么？

1. **大模型应用开发不是调API**：提示词工程和RAG才是让AI真正落地的关键，占了我最多时间也收获最大
2. **全栈能力**：从React前端到FastAPI后端到SQLite数据库到云服务器部署，完整走了一遍
3. **工程化思维**：错误处理、降级机制、配置管理、版本控制——这些"非功能需求"决定了项目是否真正可用
4. **问题解决能力**：编码问题、网络问题、防火墙问题——真实开发中大部分时间在踩坑和填坑
5. **技术选型的权衡**：每个选择都有trade-off，比如SQLite vs PostgreSQL、关键词匹配 vs 向量检索
